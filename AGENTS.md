# AGENTS.md

이 문서는 Codex가 이 저장소에서 작업할 때 따라야 할 규칙을 정의한다.

## 기본 원칙

- 모든 설명, 진행 상황, 최종 답변은 한국어로 작성한다.
- 사용자는 초보자일 수 있으므로, 어려운 용어는 가능한 한 쉽게 풀어서 설명한다.
- 단순히 "무엇을 했다"만 말하지 말고, 필요한 경우 "왜 그렇게 했는지"도 짧게 설명한다.
- 작업 전에는 기존 프로젝트 구조와 코드 스타일을 먼저 확인한다.
- 불필요한 리팩터링이나 범위 밖 변경은 하지 않는다.

## 프로젝트 특성

이 프로젝트는 Java 17, Spring Boot, Gradle 기반의 백엔드 프로젝트이다.

주요 구성은 다음과 같다.

- `src/main/java`: Java 애플리케이션 코드
- `src/main/resources`: Spring 설정 파일
- `src/test/java`: 테스트 코드
- `build.gradle`: Gradle 의존성 및 빌드 설정
- `gradlew`, `gradlew.bat`: Gradle Wrapper 실행 파일
- `Dockerfile`: 컨테이너 실행/배포 설정

작업 시 Spring Boot, Spring Security, Spring Data JPA, PostgreSQL, JWT, Supabase, Mailgun 설정이 얽혀 있을 수 있음을 고려한다.

## 작업 전 규칙

파일을 수정하기 전에는 반드시 먼저 계획을 설명한다.

계획에는 다음 내용을 포함한다.

- 어떤 파일을 확인했는지
- 어떤 파일을 수정할 예정인지
- 왜 수정이 필요한지
- 예상되는 영향 범위
- 테스트 또는 실행 확인 방법

사용자가 명시적으로 "바로 수정해"라고 하더라도, 최소한의 짧은 계획은 먼저 말한다.

## 실행 및 테스트 확인

작업을 시작하기 전에 이 프로젝트에서 사용할 수 있는 실행/테스트 명령어를 먼저 확인한다.

우선 확인할 파일:

- `README.md`
- `build.gradle`
- `settings.gradle`
- `gradle/wrapper/gradle-wrapper.properties`
- `.github/workflows/*`
- `Dockerfile`

Windows 환경에서는 기본적으로 다음 명령어를 우선 고려한다.

```powershell
.\gradlew.bat bootRun
.\gradlew.bat test
.\gradlew.bat build
```

Linux/macOS 또는 CI 환경에서는 다음 명령어를 고려한다.

```bash
./gradlew bootRun
./gradlew test
./gradlew build
```

테스트를 실행하지 못한 경우에는 최종 답변에 그 이유를 명확히 적는다.

## 코드 수정 규칙

- 기존 코드 스타일과 패키지 구조를 최대한 따른다.
- Controller, Service, Repository, Model, Dto, Config, Util의 역할을 구분해서 수정한다.
- Controller에는 요청/응답 처리 중심의 코드를 둔다.
- Service에는 핵심 비즈니스 로직을 둔다.
- Repository에는 DB 접근 로직을 둔다.
- Model은 DB 테이블과 연결되는 JPA Entity로 다룬다.
- Dto는 API 요청/응답 데이터 전달용으로 사용한다.
- 공통 기능은 기존 Util 구조를 먼저 확인한 뒤 추가한다.

## 기존 코드 삭제 규칙

기존 코드를 삭제하거나 크게 바꾸기 전에는 반드시 이유를 설명한다.

설명에는 다음 내용을 포함한다.

- 어떤 코드를 삭제하거나 변경하려는지
- 왜 삭제가 필요한지
- 삭제하지 않고 해결할 수 있는 방법이 있는지
- 삭제로 인해 영향을 받을 수 있는 기능

사용자가 명시적으로 삭제를 요청하지 않은 경우, 불필요한 삭제는 하지 않는다.

## 설정 파일 처리

`application.properties`, `.env`, Docker, Gradle 설정 파일을 수정할 때는 특히 조심한다.

- 비밀번호, API 키, JWT 키 같은 민감 정보는 코드에 직접 넣지 않는다.
- 환경변수 이름을 바꿀 경우, 실행 방법과 배포 설정에 미치는 영향을 설명한다.
- 로컬 실행과 배포 환경이 다를 수 있음을 고려한다.

## 데이터베이스 관련 작업

JPA Entity, Repository, Service를 수정할 때는 DB 구조와 연결될 수 있음을 고려한다.

- Entity 필드 변경은 실제 DB 컬럼과 맞는지 확인한다.
- Repository 메서드 이름은 Spring Data JPA 규칙을 따른다.
- DB 마이그레이션 도구가 없는 경우, 스키마 변경이 필요한 작업은 사용자에게 먼저 알린다.

## 보안 관련 작업

Spring Security, JWT, 비밀번호, 인증/인가 관련 코드는 신중하게 수정한다.

- 인증 없이 접근 가능한 API 목록을 변경할 때는 이유를 설명한다.
- 비밀번호 검증, 암호화, JWT 생성/검증 로직을 바꿀 때는 영향 범위를 설명한다.
- 보안상 위험할 수 있는 변경은 대안을 함께 제안한다.

## 초보자 친화 규칙

최종 답변에는 다음을 포함하도록 노력한다.

- 수정한 파일 목록
- 무엇이 바뀌었는지 쉬운 설명
- 실행 또는 테스트 명령어
- 테스트 결과 또는 실행하지 못한 이유
- 다음에 확인하면 좋은 점

가능하면 "이 명령어를 실행하면 됩니다"처럼 바로 따라 할 수 있는 형태로 안내한다.

## 금지 사항

- 사용자 허락 없이 대규모 리팩터링을 하지 않는다.
- 사용자 허락 없이 unrelated 파일을 수정하지 않는다.
- 기존 코드를 되돌리거나 삭제하지 않는다.
- 민감 정보를 코드에 직접 작성하지 않는다.
- 테스트 실패를 숨기지 않는다.
- 파일을 수정했는데 수정 사실을 말하지 않고 넘어가지 않는다.
